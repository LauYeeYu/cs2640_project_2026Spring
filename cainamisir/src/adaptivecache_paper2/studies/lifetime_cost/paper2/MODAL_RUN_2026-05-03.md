## What we learned

1. **Pipeline works end-to-end on Modal.** Image build + overlay apply +
   v3 Phase 1 capture all reproduce in a clean container. The overlay
   apply step needed `find ... -name __pycache__ -delete` after the
   rsync because pip pre-compiles vllm with UNCHECKED_HASH .pyc mode —
   stale bytecode caused a `TypeError: missing compacted_token_counts`
   on multi-step requests.

2. **The runner had a state-mutation bug.** `messages = list(task.messages_init)`
   was a shallow copy. Policies mutate message dicts (memento, recalled_step,
   prior_memento) — those mutations leaked back to the shared task object.
   In multi-variant bakes, variants 2..N saw the mementoed state from
   variant 1, so they SKIPPED compaction (already mementoed) and
   produced misleadingly-fast trajectories. Fix: `copy.deepcopy(task.messages_init)`
   in `run_task`. First bake (run 1777779290) was poisoned by this; this
   re-run (run 1777780585) is clean.

3. **Resolve rate is 0/3 across all variants on pytest-7490 at H100/T=0.6.**
   Different from our local Blackwell baseline (which got 1-2/3). Either
   numeric differences (different attention backend / kernel selection)
   or pytest-7490 just doesn't resolve at this setting. Need to broaden
   the task set before drawing conclusions — pytest-7490 alone is too
   noisy.

4. **Clean A/B (lru vs lru-append, seed=1, where recall actually fires):**
   lru = 33s, lru-append = 36s. Same compactions (4), same recalls (2).
   Append is +3s SLOWER, not faster.

   The intuition: append doesn't shift suffix positions, so prefix cache
   should hit better. But the addendum adds tokens to every subsequent
   chat call, and that tax (more tokens to process per step) outweighs
   the cache savings on this short trajectory. Append's cache win likely
   dominates only on LONGER trajectories with MORE recalls, where the
   re-prefill cost of inplace's cliff multiplies.

5. **Embedding vs embedding-append diverged completely** (8s vs 72s on
   seed=1, 8s vs 87s on seed=2). The trajectories are different — likely
   numeric differences from prefix-cache state across cells cause
   different sampling at T=0.6 even with same seed. Not a clean A/B.

## Open questions for next session

1. **Does append win on longer trajectories?** Need a task that triggers
   ≥5 recalls, where the cliff cost of inplace can compound. pytest-7490
   only gets 2 recalls per cell.

2. **Why does Qwen3-30B not resolve pytest-7490 on H100 when it did on
   Blackwell?** Numerics differ across kernels; need to investigate
   FlashInfer vs FlashAttention or fp16 vs bf16.

3. **Same-seed reproducibility across cells.** Even with `model._default_seed
   = seed`, cells diverged. The vLLM prefix cache state across cells may
   be the cause. To get clean A/B, may need to reset the engine between
   cells (kills throughput) OR drop temperature to 0.
