### v3 Phase 3b — Scheduler API + worker hook

- `KVRestoreOp(memento_id, request_id, target_block_ids)` dataclass
  alongside `CaptureOp` in `block_masking/memento_store.py`.
- `Scheduler.queue_kv_restore(request_id, memento_id) → (ok, message)`:
  - Looks up StoredMemento metadata in the SCHEDULER-side store
    (auto-stashed at capture time, cpu_kv=None).
  - Allocates `len(physical_block_ids)` blocks via
    `kv_cache_manager.block_pool.get_new_blocks`.
  - Appends KVRestoreOp to `pending_kv_restore_operations[request_id]`.
- `SchedulerOutput.kv_restore_operations: dict[req_id, list[KVRestoreOp]]`
  carries through to worker each step.
- `gpu_model_runner._execute_kv_restore_operations`: consumer that
  calls restore_pinned_kv per op. Runs AFTER capture (don't overwrite
  fresh bytes), BEFORE copy_ops + model forward (so attention reads
  spliced slots).
- `mask_token_span(capture_for_memento_id=...)` auto-stashes
  metadata-only `StoredMemento` on scheduler-side store at capture
  time, so `queue_kv_restore` can find block counts without a worker
  round-trip.
- `tests/test_v3_phase3b.py` — 4 unit tests with mock block_pool:
  unknown memento_id, allocate-and-queue, OOM, multi-restore-per-req.

## Where v3 stands functionally

```
chat() N
  ├─ capture fires (block_masking processor)
  │     ├─ scheduler.mask_token_span(capture_for_memento_id="auto:R:S:E")
  │     │     ├─ stash metadata-only StoredMemento in scheduler store
  │     │     └─ queue CaptureOp -> SchedulerOutput.kv_capture_operations
  │     └─ worker._execute_kv_capture_operations
  │           └─ index_select GPU blocks → pinned CPU → MementoStore.stash
  └─ chat returns

 ... (some calls later) ...

chat() N+K
  ├─ [Phase 3c GAP] policy needs a way to fire queue_kv_restore from outside
  │   — currently scheduler.queue_kv_restore is internal API; no path from
  │   the chat() entrypoint or the policy.
  │
  ├─ scheduler.queue_kv_restore("req_N+K", "auto:R:S:E")  # if we could call this
  │     ├─ allocate fresh blocks via block_pool
  │     └─ queue KVRestoreOp -> SchedulerOutput.kv_restore_operations
  │
  ├─ worker._execute_kv_restore_operations
  │     └─ restore_pinned_kv: GPU index_copy_ from CPU pinned
  │
  └─ [Phase 3c GAP #2] the restored blocks are on GPU but NOT spliced
      into the request's block_table at the obs's chronological positions.
      Without this, the attention kernel doesn't read them — they sit there
      as orphaned KV.
```

## Phase 3c — what it actually requires

The two gaps above are both "Phase 3c". Three implementation paths:

**Path A: prefix-cache hash registration** (most elegant)

When restore_pinned_kv runs, ALSO register the GPU blocks with vLLM's
prefix cache under the right hash (`h(parent_block_hash, obs_token_ids)`).
The next request's natural prefix lookup finds them → free splice.

- Pro: leverages existing mechanism; no per-request block_table editing.
- Con: requires knowing `parent_block_hash` at capture time. Stable in
  our policy (oldest-first compaction → prefix-up-to-P monotone after
  P is mementoed) — but we'd need to capture the parent hash alongside
  the KV bytes.

**Path B: direct block_table splice** (most invasive)

New `Request.injected_blocks: list[(start_logical, end_logical, block_ids)]`
field. Scheduler at request-start respects this when building block_table.
slot_mapping in gpu_model_runner reads injected blocks for those positions.

- Pro: explicit and obvious.
- Con: ~300 LOC across scheduler, request, kv_cache_manager,
  gpu_model_runner.

**Path C: NewRequestData splice** (medium invasive)

NewRequestData (sent to worker for each new request) gains
`injected_blocks` list. Worker, when initializing a request, allocates
the injected block IDs into the request's block_table at the right
indices.

- Pro: localized to NewRequestData + worker; doesn't touch scheduler
  internals.
- Con: scheduler still needs to know injected blocks to compute
  num_computed_tokens correctly.

Recommendation: **Path A** (prefix-cache registration) if vLLM's
prefix-cache API allows manual block insertion. Otherwise **Path C**
(NewRequestData splice).

## Phase 3d — outside-engine recall API

The policy needs to fire `queue_kv_restore` from outside the engine.
Options:

1. **New env-side API**: `MementoVLLMModel.queue_recall(memento_id)`
   that buffers the request; `chat()` flushes via a worker RPC.
2. **Prompt-encoded marker**: when the prompt has a special pattern
   `<|recall|>memento_id<|/recall|>`, the engine intercepts and fires
   `queue_kv_restore` automatically. Cleanest end-state.
3. **Direct scheduler access** (V1 single-process only): policy gets a
   reference to the scheduler. Doesn't work for multi-process.

Option 2 is the way — it composes with normal chat() flow.
