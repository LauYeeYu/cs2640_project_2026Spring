## TL;DR — three mechanisms, three states

| Mechanism | What it does | Cost shape | State |
|---|---|---|---|
| **v1 (inplace)** | Clear `memento` field → renderer drops in full obs at original chronological position. Tokens added in the middle shift the suffix → prefix cache misses past that point. | Cliff (~$0.10–0.15 per recall, Paper-1-style) | ✓ shipped, multi-seed bake done on pytest-7490 |
| **v2 (append)** | Leave `memento` set; push synthetic user message at the tail with `[recalled, obs_id=N]\n<full obs>`. Prefix cache hits everything before the addendum. | Cheap (~$0.001 — only the addendum prefills) | ✓ code shipped; **GPU bake pending** (blocked on shared GPU) |
| **v3 (KV-fork)** | Capture obs KV blocks at compaction time → CPU pinned memory. On recall, splice them back into a future request's block table at chronological positions. Bypass hash-based cache lookup. | Free in principle (CPU↔GPU H2D only) | ✓ Phase 1 (capture half) shipped; **GPU validation + Phase 3 (splice) pending** |

## What shipped this session (2026-05-02)

### Append-only recall (v2)

- **`paper2/policy/memento_policy.py`**: new `recall_mode={'inplace','append'}`
  knob (default `inplace` for backwards compat). `maybe_recall` in
  `append` mode leaves the original mementoed message untouched and
  pushes a new user message at the end of `messages` with the full obs
  text, marker fields (`_recall_marker`, `_recall_target_idx`), and a
  human-readable preamble `[recalled, obs_id=N]`.
- **`paper2/validate_recall.py`**: variant grammar `<base>-<mode>`.
  Examples: `off`, `lru`, `lru-append`, `embedding`, `embedding-append`.
- **`paper2/tests/test_recall.py`**: 9 unit tests, 2 new for append
  branch. All green.
- **`paper2/V2_DESIGN.md`**: design doc — why prompt-layer append is
  the cheap-recall mechanism and how the cache analysis works out.

## Roadmap

### Immediate (next GPU window)

1. Run `validate_recall.py` with the 5-variant grid → confirm v2 append
   wins on `cached_tokens` and wall, holds resolve. (~30 min wall.)
2. Run `smoke_v3_capture.py` → confirm Phase 1 capture chain fires and
   bytes land. (~5 min wall + engine warmup.)

### Short-term (1–2 sessions)

3. **Phase 1 worker↔scheduler surface**: if the smoke test confirms the
   worker captures into its own process MementoStore (likely), add a
   shim that surfaces `memento_ids` and `total_cpu_bytes` to the
   scheduler/policy via the existing engine RPC channel. Without this,
   the policy can't enumerate what's captured.
4. **Phase 1 multi-task bake**: with append working, run v0 (no recall),
   v1 inplace, v2 append, on 4–8 SWE-bench tasks × 3 seeds. Compare
   wall, cached_tokens, resolve.

### Medium-term (Paper 2's actual contribution)

5. **Phase 2** (optional): `block_pool` LRU exemption for memento_id-
   tagged blocks → keeps captured KV on GPU when budget allows; skip
   the H2D round-trip on recall. Defer until Phase 3 is working and we
   measure CPU↔GPU recall cost.
6. **Phase 3 (the actual win)**: recall splice. Patch
   `kv_cache_manager.get_computed_blocks` to accept an explicit
   `(memento_id, target_logical_range)` injection per request — bypass
   hash lookup for that range. Patch slot_mapping in `gpu_model_runner`
   to interleave injected slots with freshly-prefilled slots. Patch
   adapter to render full obs in chronological position when policy
   declares a recall AND pass the recall metadata to the engine via
   `prompt_token_ids` extras.
   - Surface area: ~500 LOC across scheduler, kv_cache_manager,
     gpu_model_runner, memento_vllm.py.
   - Validity invariant the policy must enforce: never re-render a
     message older than a captured obs's position. The current
     oldest-first compaction satisfies this naturally — once obs at P
     is captured, no compaction earlier than P will fire (those
     positions are already mementoed).
   - Risk: stored KV is computed against the prefix-as-it-was-then.
     Reusing it later means accepting "stale historical truth"
     (V3_DESIGN.md, "The fundamental constraint"). Empirical question
     whether quality holds; mitigation in V3 risk register.

### Long-term

7. **Quality study**: replay a known prompt with stale vs fresh KV;
   measure logit divergence at the next token. Establishes the
   bound the model can tolerate.
8. **Real eval grid**: 4 variants (none, consumption_evict, memento,
   memento+recall — pick best mode) × 8 tasks × 3 seeds. Report
   resolve@k, GPU-seconds per resolved, prompt-tokens per resolved,
   recall accuracy.
