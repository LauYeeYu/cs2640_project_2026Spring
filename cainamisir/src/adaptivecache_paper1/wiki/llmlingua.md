# LLMLingua & LongLLMLingua: Prompt Compression for Accelerated LLM Inference

**Authors:** Huiqiang Jiang, Qianhui Wu, Chin-Yew Lin, Yuqing Yang, Lili Qiu (LLMLingua); + Xufang Luo, Dongsheng Li (LongLLMLingua)  
**Affiliation:** Microsoft Corporation  
**LLMLingua Date:** EMNLP 2023 (arXiv October 2023)  
**LongLLMLingua Date:** 2024 (arXiv October 2023, revised August 2024)  
**arXiv:** 2310.05736 (LLMLingua), 2310.06839 (LongLLMLingua)  
**Source files:** `raw/papers/llmlingua.pdf`, `raw/papers/longllmlingua.pdf`

## Problem

LLM inference is increasingly bottlenecked by long prompts — ICL demonstrations, retrieved documents, and multi-turn conversation histories push prompts to tens of thousands of tokens. Two structural issues compound the computational cost:

1. **Redundancy:** Natural language is highly redundant; many tokens carry near-zero marginal information and can be removed without loss of downstream task performance.
2. **Position bias ("lost in the middle"):** Even if the right tokens are present, their position in the prompt determines whether the model uses them. Tokens in the middle of long prompts are attended to far less reliably than tokens at the beginning or end.

Prior compression methods (Selective-Context) compute token self-information with a small model and drop low-information tokens, but ignore (a) inter-token dependencies (the independence assumption), (b) alignment between the small compressor model and the target LLM, and (c) question-relevance of tokens when the prompt contains multiple documents.

## Proposed Methods

### LLMLingua (EMNLP 2023) — Coarse-to-Fine Prompt Compression

**Core idea:** Use a small language model (Alpaca-7B) to assess token importance via perplexity, then compress in three stages:

**1. Budget Controller:** Assign differential compression ratios to prompt components. Instructions and questions get lower compression (τ_ins=0.85, τ_que=0.9); demonstrations get higher compression. At high compression ratios, perform coarse-grained sentence/demonstration-level dropout first to maintain linguistic integrity, rather than token-level dropout which breaks reasoning chains. Demonstrations are ranked by perplexity and dropped in order until the budget for the demonstrations component is met.

**2. Iterative Token-level Prompt Compression (ITPC):** Segment the prompt and compress iteratively. Each segment's compressed output is prepended to the next segment's context before computing perplexity thresholds. This accounts for conditional dependencies between tokens — a key improvement over Selective-Context's independence assumption. Tokens with perplexity above a dynamic threshold γ_j are retained.

**3. Distribution Alignment:** Fine-tune the small compressor model M_s on instruction data generated by the target LLM (Alpaca dataset fine-tuning on GPT-3.5-Turbo outputs). This reduces the distribution gap between the small model's perplexity estimates and the target LLM's actual sensitivity, improving compression accuracy.

**Formulation:** The objective is to minimize KL(P(G̃|x̃), P(G|x)) — the compressed prompt's output distribution should match the original prompt's output distribution.

### LongLLMLingua (2024) — Question-Aware Long-Context Compression

LongLLMLingua extends LLMLingua to address three problems specific to long contexts: higher cost, performance reduction, and position bias.

**1. Question-Aware Coarse-Grained Compression:** For each document x_k^doc, compute the importance metric r_k = -1/N_c Σ log p(x_i^{que,restrict} | x_k^doc). This measures how well document k predicts the question (conditional perplexity of the question given the document), rather than the document's own perplexity. Documents that "explain" the question better score higher. A restrictive statement ("We can get the answer to this question in the given documents") is appended to reduce hallucinations. This metric outperforms all retrieval baselines (BM25, OpenAI embedding, BGE-large, Cohere-Rerank) on recall.

**2. Document Reordering:** After coarse-grained scoring, reorder documents by their importance score r_k before fine-grained compression. More relevant documents are placed at the beginning and end of the context, exploiting the primacy and recency effects identified by Lost in the Middle. Ablation confirms this is effective even on timeline-related tasks.

**3. Question-Aware Fine-Grained Compression via Contrastive Perplexity:** For each token x_i in the retained documents, compute:
`s_i = perplexity(x_i | x_{<i}) - perplexity(x_i | x^{que}, x_{<i})`
This "contrastive perplexity" measures how much the question reduces the token's uncertainty — a token highly predicted by the question is question-relevant. Mathematically, s_i ≈ p(x^{que} | x_i, x_{<i}), i.e., the conditional likelihood of generating the question given the token. This is equivalent to conditional pointwise mutual information. Tokens with high contrastive perplexity cluster near the ground-truth document; raw perplexity does not.

**4. Dynamic Compression Ratio:** Allocate lower compression (more tokens retained) to documents with higher r_k scores. Uses a linear scheduler to bridge coarse and fine granularity.

**5. Subsequence Recovery:** After the LLM generates a response from the compressed prompt, recover original token sequences by finding maximum common subsequences between the compressed prompt and original prompt. This restores proper entity names and numbers that were corrupted by tokenizer boundary effects during compression.

## Key Results

**LLMLingua:**
- **20× compression** on GSM8K with only 1.5 points EM drop (vs. full 8-shot CoT: 78.85%)
- **5× compression** on GSM8K with higher performance than full-shot (79.08% vs. 78.85%)
- **3.3×–9× speedup** on Arxiv summarization and ShareGPT conversation tasks
- **1.7×–5.7× end-to-end latency reduction** on V100, depending on compression ratio
- **Cost reduction:** $4.7 → $0.5 for GSM8K; $8.0 → $4.8 for BBH using GPT-3.5-Turbo
- **Ablation:** Every component contributes; iterative compression is most important (7-point EM gap without it); distribution alignment adds 0.56 EM

**LongLLMLingua:**
- **+21.4% accuracy** on NaturalQuestions (20 documents) at 4× compression with GPT-3.5-Turbo
- Compressed prompts **outperform original prompts** — noise removal at 4× gives better performance than the full 2,946-token original
- **94.0% cost reduction** on LooGLE long dependency benchmark
- **1.4×–2.6× latency speedup** at 2×–6× compression for 10K-token prompts
- **LongBench:** 48.3 average (vs. 44.0 original prompt) at 6× compression
- **ZeroSCROLLS:** 32.7 average (vs. 32.5 original) at 6× compression
- **Document reordering alone** flattens the U-shaped position-bias curve — LongLLMLingua at 4× with reorder achieves roughly uniform accuracy across all 20 document positions

## Limitations

LLMLingua produces grammatically distorted output (tokens dropped mid-sentence) that is interpretable by LLMs but not humans. At 25–30× compression, all methods show significant drops. LongLLMLingua is question-aware and thus per-query — the same context must be re-compressed for different questions, preventing context caching. This is explicitly noted as a limitation; the authors suggest extending to task-aware compression to enable reuse. LLMLingua also increases computation by 2× compared to vanilla inference (small model pass + LLM pass vs. just LLM pass).

## Relationship to AdaptiveCache

LLMLingua and LongLLMLingua are the closest competitors to AdaptiveCache in the prompt compression space, and they directly illustrate both the opportunity and the structural gap that AdaptiveCache addresses.

**Points of contact:**
- **Perplexity-based importance = complementary signal to AdaptiveCache's attention signal:** LLMLingua uses small-model perplexity to identify low-information tokens. AdaptiveCache uses attention scores (H2O/SnapKV style) to identify low-attention tokens. These two signals capture different aspects of token importance. LLMLingua's contrastive perplexity (question-conditional PMI) could be used as an additional scoring signal within AdaptiveCache's compaction step — tokens that score high on both attention importance AND contrastive perplexity are the strongest stable prefix candidates.
- **LongLLMLingua's reordering validates AdaptiveCache's core hypothesis:** LongLLMLingua's document reordering step — placing high-importance documents at the beginning and end of context — empirically validates AdaptiveCache's layout hypothesis. The reordering ablation in LongLLMLingua shows that layout matters for performance beyond just which tokens survive. AdaptiveCache generalizes this from document-level reordering to token-level layout optimization with prefix KV-cache awareness.
- **LongLLMLingua's contrastive perplexity as an AdaptiveCache signal source:** AdaptiveCache's inter-step compaction needs to identify which prior context tokens are most relevant to the current agent task. LongLLMLingua's contrastive perplexity — measuring how strongly each token predicts the current query/question — is a natural implementation of this. AdaptiveCache can compute contrastive perplexity at each step boundary using the newly-generated observation as the query x^{que} to score prior context tokens.
- **Question-aware coarse-to-fine compression structure:** LongLLMLingua's two-level compression structure (document-level importance scoring → token-level compression within retained documents) is directly applicable to AdaptiveCache's promotion decision: first identify which agent observations/trajectory segments are most relevant (coarse), then identify which tokens within those segments belong in the stable prefix (fine).
- **Cost reduction numbers validate the business case:** LLMLingua/LongLLMLingua show 70–94% cost reductions on realistic tasks. These numbers directly quantify the potential savings from AdaptiveCache's approach applied to similar agent contexts. AdaptiveCache's prefix-caching motivation aligns with LLMLingua's cost reduction motivation.

**Key differences from AdaptiveCache:**
1. **Prefix-cache-invalidating:** Both LLMLingua and LongLLMLingua drop tokens from their original positions, producing a compressed prompt with different byte content than any prior prompt. This invalidates all downstream prefix KV-cache reuse — the serving system cannot recognize any shared prefix between consecutive calls. AdaptiveCache specifically preserves prefix layout for cross-step KV-cache reuse.
2. **Not persistent across agent steps:** LLMLingua re-compresses each prompt from scratch for each LLM call. There is no mechanism to accumulate importance evidence across multiple agent steps. AdaptiveCache's stable prefix grows more stable across steps as tokens accumulate attention evidence.
3. **Grammatically incoherent output:** LLMLingua's token-level dropping produces partially readable text (numbers and words truncated mid-token). This works for black-box LLM inference but would not work as the actual context in an agent's memory. AdaptiveCache preserves complete tokens and their ordering within selected segments.
4. **Requires small LM overhead:** LLMLingua requires a separate 7B model to compute perplexity for every compression event. AdaptiveCache uses attention weights already computed during the LLM's forward pass — zero additional model overhead.
5. **Per-query compression (LongLLMLingua limitation):** LongLLMLingua explicitly acknowledges it cannot cache compressed contexts because different queries need different compressions. AdaptiveCache's attention-signal-based approach uses the same mechanism but can build a stable prefix that persists across steps because the importance ranking stabilizes over time.

**AdaptiveCache synthesis:** AdaptiveCache should treat LLMLingua/LongLLMLingua as the primary empirical baseline demonstrating what token-level prompt compression can achieve: 20× compression ratios with performance preservation or even improvement. The critical gap is that both LLMLingua papers produce prefix-invalidating compressed outputs. AdaptiveCache's unique contribution is ensuring that compressed output is arranged as a valid, stable prefix for KV-cache reuse. LongLLMLingua's document reordering insight and contrastive perplexity scoring are directly adoptable as sub-components of AdaptiveCache's importance signal, with the layout step added on top.
